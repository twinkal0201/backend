# Cookies and Sessions

## Introduction

HTTP is a **stateless protocol**, which means the server does not remember information about previous requests. Every request is treated as a completely new request.

For example, suppose a user logs into Instagram. The server verifies the username and password and allows access to the account. Later, when the user likes a post, uploads a photo, or sends a message, the server must know which user is performing these actions.

Without a mechanism to remember the user, the server would have to ask for login credentials on every request. This is solved using **Sessions** and **Cookies**.

---

## Working of Sessions and Cookies

### Step 1: User Login

The user enters a username and password and sends a login request to the server.

```
POST /login
username = john
password = 123
```

The server verifies the credentials.

---

### Step 2: Session Creation

After successful authentication, the server creates a unique Session ID.

Example:

```
Session ID = ABC123XYZ
```

The server stores information associated with this Session ID.

```
ABC123XYZ -> User John
```

This information stored on the server is called a **Session**.

---

### Step 3: Cookie Creation

The server sends the Session ID to the browser.

```
Set-Cookie: sessionId=ABC123XYZ
```

The browser stores this value automatically.

This small piece of data stored in the browser is called a **Cookie**.

---

### Step 4: Future Requests

When the user performs another action, such as liking a post, the browser automatically sends the cookie along with the request.

```
POST /like-post

Cookie: sessionId=ABC123XYZ
```

---

### Step 5: User Identification

The server receives the Session ID and searches for it in its session storage.

```
ABC123XYZ -> User John
```

The server identifies the user as John and processes the request without asking for login credentials again.

---

## Cookie

A cookie is a small piece of data stored in the user's browser. It is automatically sent to the server with future requests.

Example:

```
sessionId = ABC123XYZ
```

### Advantages of Cookies

* Maintains user state.
* Reduces repeated authentication.
* Improves user experience.

---

## Session

A session is data stored on the server that keeps track of a user's activity and login state.

Example:

```
Session ID : ABC123XYZ
User       : John
Login Time : 10:00 AM
```

### Advantages of Sessions

* More secure because data is stored on the server.
* User information is not exposed to the client.
* Easy to invalidate during logout.

---

## Relationship Between Cookie and Session

```
Browser                         Server
--------                        --------

Cookie
sessionId=ABC123XYZ  ------->   Session
                                ABC123XYZ -> John
```

The cookie stores the Session ID, while the session stores the actual user information. Together, they help the server recognize authenticated users across multiple requests.

---

## Conclusion

Cookies and Sessions are used to maintain user authentication and state in web applications. The server creates a session after successful login and sends the Session ID to the browser as a cookie. The browser automatically includes this cookie in future requests, allowing the server to identify the user without requiring repeated logins. The time period between login and logout (or session expiration) is called a session.
